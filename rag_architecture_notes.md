# 🎓 RAG 架構設計筆記：離線 FAQ 訓練 vs 實時查詢擴展

本筆記記錄了本智慧檢索系統在處理「地端推論延遲（反應速度）」與「跨語言檢索（回覆準確度）」時的核心架構權衡。

---

## 🔍 RAG 雙路檢索與雙模架構的核心觀念

在我們的系統中，有兩個非常關鍵的核心功能：
1. **「離線 FAQ 自訓練」**：將冷冰冰的 PDF 書面官方條文，轉換為大量（460+ 筆）學生日常口語問答，並預先計算好 Embedding 快取。
2. **「實時查詢擴展 (Query Expansion)」**：在提問當下，呼叫大語言模型（LLM）將使用者的問句即時改寫、擴充並翻譯成英文。

---

## ⚖️ Offline Training vs. Online Inference 的權衡

| 評估維度 | 離線 FAQ 自訓練 (Offline Training) | 實時查詢擴展 (Online Inference) |
| :--- | :--- | :--- |
| **運算時機** | 離線一次性計算（使用自訓練腳本） | 每次提問時在線即時計算 |
| **時間開銷** | **0 毫秒**（提問時直接進行資料庫比對） | **3~5 秒**（提問時需先跑一次 LLM 生成） |
| **跨語言對齊** | 預先完成（口語問題為中文，對齊英文原文） | 即時翻譯（中文即時翻譯為英文） |
| **主要功能** | 提升口語化提問的語意命中與原文帶出 | 彌補冷啟動文件與處理複雜生僻提問 |

---

## 🎯 結論：為什麼「加速模式」能與 FAQ 訓練互補？

當我們將離線 FAQ 訓練做得非常飽滿（例如全庫 463 筆 FAQ）時，**FAQ 資料庫本身就已經扮演了「預先計算好的靜態擴展詞庫」角色**。

因此：
1. **系統預設開啟「⚡ 查詢加速模式」是最佳實踐**：
   因為絕大多數的口語問題都能在 0 毫秒內直接命中 FAQ，完全不需呼叫 LLM 進行即時查詢擴展，地端速度因此能提升 **3~5 秒（秒級回應）**。
   
2. **「非加速模式（即時查詢擴展）」依然是不可或缺的防護網**：
   * **解決冷啟動 (Cold Start)**：當有全新、尚未進行 FAQ 訓練的 PDF 文件加入 `data/` 時，即時查詢擴展是撈到英文 PDF 內容的唯一手段。
   * **處理極端提問 (Corner Cases)**：應付極度複雜、冷門、離線 FAQ 沒有覆蓋到的長提問。

## 📌 未來優化指引

為求極致的性能，本系統採用的**混合策略 (Hybrid Strategy)**：
* 系統中實施了**「地端零延遲雙語擴展」**，針對請假問題直接用規則詞庫做 0 毫秒靜態擴充；非請假問題直接旁路（Bypass）跳過。
* 未來若加入大量新 PDF 檔案，建議先執行一次離線 FAQ 訓練腳本（`generate_faq_dataset.py`），以確保即使在「⚡ 查詢加速模式」下也能擁有 100% 的口語檢索精準度。
